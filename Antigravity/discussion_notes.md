# Antigravity 框架深度討論紀錄

> 討論日期：2026-04-10  
> 性質：框架設計問題診斷 + 改進方向腦力激盪  
> 狀態：待後續實施

---

## 一、已確認的根本問題

### P1：框架設計對象錯位
**問題**：Antigravity 的透明化機制（跨語系解析 Panel、系統準備清單、足跡收據）是為「技術型總監」設計的，假設閱讀者能驗證 AI 的中間過程。  
**現實**：總監是零技術背景，只能驗證最終結果（功能對不對、方向對不對），無法評估程式碼差異或技能是否正確載入。  
**影響**：透明化輸出對總監而言是噪音，且消耗大量 Tokens 與上下文空間。

### P2：技能觸發機制是概率性的，不是確定性的
**問題**：技能的 `Use when:` 描述依賴 AI 自行語義匹配，決定何時載入哪個技能——這是「建議式」而非「命令式」。  
**根因**：工作流 YAML 中的 `required_skills` 欄位只是**聲明**，沒有任何機制強制 AI 在開始執行前讀取它們。AI 可以跳過，沒有攔截。  
**影響**：技能命中率不穩定，尤其是認知負荷高的任務或繁體中文輸入時更容易漏失。

### P3：技能內容格式與使用期望不符
**問題**：現有技能大多是「參考文件」格式（知識性描述），但被期望以「執行程序」方式運作（步驟式指令）。  
**補充說明**：技能鍛造系統（`skill-factory`）已設計三種風格：命令式（imperative）、混合式（hybrid）、引導式（guided），設計本身是正確的。問題在於工作流端沒有強制執行這個分類——技能寫得再精準，不被載入也沒用。

| 現況（參考文件） | 期望（執行程序） |
|---|---|
| 說明 AI 在什麼情況下使用 | 命令 AI 第一步做什麼、第二步做什麼 |
| AI 自行判斷何時讀取 | 工作流強制在任何步驟前先讀 |
| 命中率靠語義匹配 | 命中率靠工作流鎖定 |

### P4：CLI 子代理的 SOP 有人工阻塞點
**問題**：現有 `cli-delegation-sop.md` 的 Step 7 要求「總監通知主腦 CLI 已完成」，總監變成人肉訊號燈。  
**已確認事實**：
- `gemini -p "..." --yolo` Headless 靜默模式**技術上可行**（已測試 v0.37.0）
- `command_status` 輪詢也可行，但消耗 Tokens，且有哲學上的浪費感
- 根本限制：AI 驅動力來自使用者輸入，無法自主等待後繼續
- 工作流中確實已有 CLI 委派設計，問題不是「沒有」，而是「品質不足」

**背景補充**：
- 曾誤以為「工作流沒有 CLI 委派」，是錯誤的
- `/08_audit_index` 的 §3.5 Step 1 有明確的「無條件強制委派 CLI」指令
- `delegation-strategy` 技能下有完整的 `references/cli-delegation-sop.md`

### P5：主腦自我驗證存在「確認偏誤」污染
**問題**：主腦在同一上下文視窗內切換批評者角色驗證自己——帶著完整推理記憶，不可能客觀。  
**CLI 子代理的正確角色**：CLI 每次啟動是乾淨的上下文，沒有主腦的推理記憶，天然實現「認知隔離」，這才是真正的獨立驗證。  
**現況缺口**：CLI 目前定位為「分析工具」（read-only analytical subagent），未被設計成「獨立驗證者」。

### P6：上下文腐敗（Context Rot）無法被使用者感知
**問題**：長對話中 AI 逐漸遺忘早期的前提與約束，但使用者不知道 AI 忘了什麼，錯誤在不知情的狀況下持續疊加。  
**現有緩解**：`/11_handoff` 工作流，但介入太晚（等到快斷線才處理）。

### P7：討論模式完全缺乏技能／記憶觸發路徑（核心新發現）
**問題**：技能系統的所有觸發機制（`Use when`、`required_skills`）都假設 AI 在工作流執行模式。純討論、探索、即興問答的場景沒有任何觸發路徑，只能靠 AI 語義自發匹配——命中率天生低。  
**更深層的問題**：討論模式需要的不是「技能（執行程序）」，而是「記憶（專案脈絡）」。兩者職責完全不同：
- **技能** = 如何執行特定任務的步驟程序（execution procedure）
- **記憶** = 關於這個專案的累積知識與決策（persistent context）

過去設計混淆了兩者，導致在討論模式下，既沒有正確的技能命中，也沒有自動的記憶加載。

### P8：記憶系統的定位被設為被動附屬（框架階層問題）
**問題**：框架現有階層順序為：`.gemini → AGENTS → rules → workflows → SKILL → 記憶系統`
**實際上**：記憶系統對 AI 執行品質的影響力應該超越工作流，因為：
- 工作流只在被觸發的時間段內有效，結束後消失
- 記憶系統承載跨對話、跨工作流的連續性——AI「認識這個專案」的唯一基礎
- 沒有記憶的工作流是盲目的；沒有工作流的記憶，AI 至少能做出合理判斷

**正確的影響力排序應該是**：

```
.gemini（母層） → AGENTS（哨兵） → rules（底層規範）
                                         ↓
                                   記憶系統（專案真相/狀態）
                                         ↓
                                   SKILL（執行知識）
                                         ↓
                                   workflows（執行程序）
```

## 二、記憶系統深度分析

### 架構評估（正確的部分）

記憶系統基於 `SKILL.md` 格式建構的設計是一個**真正的架構創新**，核心洞見是：利用 IDE 能強制鎖定 SKILL 具體位置的特性，把「記憶在哪？」這個不確定問題，轉換成「SKILL.md 在可預測路徑下」的確定性結構。

這讓 MCP `cartridge-system` 能接入，實現：
- ✅ 統一格式（SKILL.md）讓 MCP 能追蹤所有記憶卡
- ✅ 樹狀拓樸（`memory-arch`）讓記憶有層級組織（最深 4 層）
- ✅ 兩步驟提交義務（`write_to_file` → `memory_commit`）確保寫入完整性
- ✅ 過期追蹤（staleness 指數）讓記憶不會靜默腐敗
- ✅ `_system/SKILL.md` 作為全域系統記憶的容器設計正確

### 架構缺口（缺失的一塊）

**記憶系統被設計成 Pull 模型（被動），但它的角色應該是 Push 模型（主動）。**

`memory_list`、`memory_read` 都是 Pull 工具——你得先想到要呼叫它，才知道有什麼記憶。問題是：AI 在對話剛開始時，不知道這個專案有什麼記憶存在，必須先有人告訴它去看，才能看。這是一個先有雞還是先有蛋的問題。

**結論**：記憶系統建了 95% 的房子，只差最後一塊拼圖——「對話啟動就自動進入上下文」的 Push 機制。

## 三、改進方向

### D1：工作流強制技能預載閘門（優先級：🔴 最高）
在每個工作流執行起點，加入不可跳過的強制閘門。改工作流格式，立即提升所有技能命中率，不需動技能內容本身：

```
[PRE-EXECUTION GATE — 不可略過]
啟動任何步驟之前，必須依序讀取以下技能：
1. view_file: {skill-a}/SKILL.md
2. view_file: {skill-b}/SKILL.md
確認閱讀完畢後，方可進入 Step 1。
```

### D2：技能內容改為指令式格式（優先級：🟡 中）
技能鍛造（`skill-factory`）已有三種風格分類設計，方向正確。問題是**現有已建立的技能大多未達到真正的指令式密度**。建議逐步重寫，優先處理最高頻使用的技能（`memory-ops`、`code-audit`、`delegation-strategy`）。

### D3：CLI SOP Step 7 改為輕量觸發（優先級：🔴 最高）

**方案 A（主腦輪詢）**：主腦 `command_status` 輪詢，CLI 完成後自動繼續，使用者完全無感。消耗少量 Tokens。  
**方案 B（精簡通知）**：保留使用者觸發，但簡化格式為：「✅ CLI 完成後請回覆『繼續』」  
**方案 C（Headless 短任務）**：對短任務使用 `gemini -p "..." --yolo`，主腦直接讀取 stdout，無需任何 Step 7

### D4：CLI 子代理角色擴充為「獨立驗證者」（優先級：🟡 中）
在工作流完成節點，主腦將產出物打包（只含輸出，不含推理過程），透過 CLI 子代理進行獨立評估。  
**適用場景**：建構完成後邏輯驗證、記憶卡更新後一致性確認。

### D5：縮減面向總監的劇場輸出（優先級：🟢 低）
跨語系守衛雙面板、系統準備清單、足跡收據——考慮改為預設收起，或重新設計哪些輸出對零技術背景使用者真正有意義。

### D6：主動上下文壓縮觸發（優先級：🟢 低）
偵測累計 Tokens 超過閾值時，主動提示總監考慮啟動 `/11_handoff`，而非被動等待。

### P7：討論模式完全缺乏技能/記憶觸發路徑（新發現）
**問題**：技能系統的所有觸發機制（`Use when`、`required_skills`）都假設 AI 在工作流執行模式。純討論、探索、即興問答的場景沒有任何觸發路徑，只能靠 AI 語義自發匹配——命中率天生低。  
**更深層的問題**：討論模式需要的不是「技能（執行程序）」，而是「記憶（專案脈絡）」。兩者職責不同，過去設計把它們混為一談。

- 第二層（Gateway）：`_system/SKILL.md` 記錄所有模組記憶卡的範圍，AI 據此判斷是否需要讀更深的記憶

**優點**：1 次固定讀取，Token 成本可預測；把「AI 語義猜測」替換為「結構性強制讀取」。

## 四、規範層深度分析

### P9：always-on 規範內部冷熱分層不足
**問題**：`01_cross_lingual_guard` 是 always-on，每次中文輸入都觸發完整的雙面板 + 收據輸出。這個規範本身的目的正確（語意確認系統），但內部有些部分在每次對話都觸發是不必要的。  
**宏觀分層現況**：00 + 01 always-on，02–05 on-demand——宏觀正確。  
**問題粒度**：在 always-on 規範「內部」，某些輸出格式只在複雜推理時有意義，卻因為在 always-on 規範裡而每次都全量觸發。

### P10：規範確定性強制缺失（本次對話的活體證明）
**問題**：規範說「MUST do X」，但沒有機械攔截——AI 在認知負荷高或對話過長時直接跳過，沒有任何後果。  
**本次對話實證**：本輪討論進行約 1 小時後，跨語系面板和足跡收據停止出現，規則被隱性跳過，沒有任何警報或自我更正機制介入。  
**根因**：HALT 機制目前主要在工作流層，核心規範 `00_core_identity` 本身幾乎沒有 HALT 保護——它告訴 AI「必須做 X」，卻沒有定義「如果沒做 X 就停下來報告」的機制。

### 跨語系守衛的定位確認
**總監定義**：跨語系守衛 = 語意確認系統，讓總監確認 AI 有沒有正確理解意圖。這個功能不能砍。  
**可優化的部分**：

| 輸出元件 | 對總監的驗證價值 | 調整方向 |
|---|---|---|
| 語意解析面板（四層解析） | 🔴 高——這是核心功能 | 保留不動 |
| 系統準備清單 | 🟡 中——有一定確認用途 | 保留，但可瘦身 |
| 足跡收據（Turn + 工具清單） | 🟢 低——工具清單對零技術背景無法驗證 | 考慮移除工具清單，只保留 Turn 計數 |

## 五、改進方向（規範層）

### D9：核心規範加入 HALT 防護（優先級：🟡 中）
把 `00_core_identity` 裡的「MUST」語言升級為「MUST + HALT if not」：  
- 例：「修改原始碼前未建立實作計畫 → HALT 並報告」  
- 例：「跨語系面板在三輪對話內未輸出 → HALT 並自我修正」  
**目標**：把規範從「告示牌」升級為「感應器 + 鎖門機制」。

### D10：足跡收據瘦身（優先級：🟢 低）
- 移除工具清單（技術細節，零技術背景總監無法有意義地驗證）
- 保留 Turn 計數（唯一有通用驗證意義的欄位）
- 或將整個收據改為預設收起的 collapsible 區塊

## 七、給下一個 AI 的交接摘要

### 總監背景
- 零技術背景，由 AI 全權負責編程與建構
- 提供想法、視覺方向；AI 負責實作、解惑、指引
- 無法驗證程式碼差異，只能驗證最終結果
- Tokens 消耗**不是優先考量**，正確性優先

### 已確認的核心原則（本次討論）
1. **不對抗概率性，改繞開它**：把「需要 AI 判斷的節點」替換為「結構性強制步驟」
2. **記憶系統>工作流**：記憶是持久知識，工作流是短期程序
3. **AGENTS.md 是最佳 Push 機制載體**（可攜、隨框架部署、already-on）
4. **CLI 子代理的本質是認知隔離**，不是工具擴充
5. **跨語系守衛的語意確認功能保留**，收據部分可瘦身
6. **規範需要 HALT 保護**，不只是「MUST」語言

## 八、第二輪討論成果（2026-04-10 第二次對話）

> 性質：D8 設計深度討論 + D7 方向確認
> 狀態：D8 定案，D7/D1/D3/D9 待後續討論

### 本輪關鍵修正

**D3 定位修正**：
- ❌ 原理解：D3 = 去掉人工確認節點
- ✅ 正確定位：D3 = CLI SOP **品質不足**的改進（現有 SOP 寫得不夠好，導致委派結果不穩定）

### D8 設計定案：`_map` 地圖記憶卡

**核心概念轉變**：
- ❌ 原設計：在 `_system/SKILL.md` 新增 Module Memory Index 章節
- ✅ 新設計：建立獨立的 `_map` 記憶卡，職責單一——純導航索引

**為什麼要獨立？**
- `_system` 現有職責是 tech stack / 部署設定（`memory-ops §5` 定義），不應承擔導航功能
- 獨立卡讓職責清晰，維護邊界明確
- 遵循底線前綴命名慣例（同 `_system`、`_assets`、`_ghost_bin`）

**格式定案**：
```
| 模組名稱 | 一句話範圍 |
|---------|-----------|
| api-routes | API 路由定義與權限管理 |
| frontend   | 前端元件與頁面結構    |
| ...        | ...                   |
```
- 僅列 Layer 1 父卡（子卡由各父卡的 `## Relations` 管理）
- 兩欄格式（名稱 + 範圍），不放讀取時機（交由工作流閘門 D1 負責）
- 體積極精簡，每次 Push 成本固定且低

**維護機制**：
- **工作流義務**：新增/刪除 Layer 1 父卡的流程（`memory-arch §1`）強制加入「更新 `_map`」步驟
- **插件監控**：cartridge-system 偵測 `.agents/memory/` 根層結構性變動（新增/消失的 SKILL.md），觸發 `_map` 過期提醒
  - 注意：觸發條件是**結構性變動**，不是內容更新（內容更新不需要動 `_map`）
  - 插件功能調整留至後續迭代（本輪討論擱置）

**分層讀取設計**：
```
_map    → 每次對話必讀（D7 Push，成本固定且低）
_system → 需要技術基礎狀態時讀（工作流按需）
父卡    → 執行對應模組任務時讀（D1 工作流閘門指定）
子卡    → 由父卡 ## Relations 導航按需讀取
```

### D7 方向更新

### D7 最終設計定案：記憶 Push 機制（雙重觸發）

**核心機制：雙層互補設計**

| 層級 | 實施位置 | 功能 |
|------|---------|------|
| 層一（主動聲明） | 新建 `.agents/rules/06_memory_push.md` | 定義三路徑結構性探測邏輯 |
| 層二（嵌入可靠機制） | 修改 `01_cross_lingual_guard.md` 系統準備清單 | 利用面板前向聲明承諾機制，Turn=1 時強制觸發 |

**為什麼不放在 AGENTS.md？**
- AGENTS.md 語義是「哨兵訊號」（框架已初始化），不應承擔行為命令
- 獨立規範職責清晰，維護邊界明確
- 與現有 `00`/`01` always-on 規範模式一致

**`06_memory_push.md` 三路徑設計：**
```
對話啟動 → 呼叫 cartridge-system__memory_list()
├── "_map" 在清單中        → 呼叫 memory_read("_map") 載入索引
├── "_map" 不在，但清單非空 → 呼叫 memory_read("_system")（若存在）
│                               + 提示「⚠️ _map 尚未建立，建議初始化」
└── 清單為空               → 提示「📭 無記憶卡，以純對話模式繼續」
```

**為什麼不用 HALT 條件？**
- HALT = 自我監控（AI 跳過閘門時已失去意識，無法感知自己跳過）
- 改用「前向承諾機制」：在 `01` 的系統準備清單輸出時，Turn=1 MUST 在「決策與應變機制」欄位公開聲明「執行記憶啟動探測」
- 承諾是對總監可見的公開宣告，不履行等於對自己輸出違約，認知壓力比 HALT 更有效

**`01_cross_lingual_guard.md` 改動（風格 A — 輕量嵌入）：**
- 在「決策與應變機制」欄位說明下加一行：`Turn = 1 時：此欄位 MUST 包含記憶啟動探測聲明，並在面板後立即執行。`
- 選擇輕量而非新增 Gate Block，避免讓 `01` 規範繼續加重

**遷移策略：**
- 透過 `install.ps1` 推送至所有已部署子專案（同 AGENTS.md 模式）
- 新專案自動包含此規範

### D1 設計定案：工作流強制技能預載閘門

**D1 解決的層級：** 輕量感知（IDE 已處理）→ 深度讀取（D1 inline Gate 補足）

### D9 設計定案：核心規範 HALT 防護

**設計原則**：HALT 只保護「跳過後造成不可逆損傷」的節點，不大量使用（加太多 = 注意力稀釋）

**✅ 升級項目 1：實作計畫必須先於原始碼寫入**
- 無聲損傷：🔴 最高 — 跳過 = 總監失去審閱權，退版成本高
- HALT 觸發條件：偵測到即將執行 `write_to_file` 修改原始碼，但 `implementation_plan.md` 尚未存在或未經 `notify_user` 送審 → HALT
- 放置位置：`00_core_identity.md` §3 Lifecycle Protocol，PLANNING 步驟後增設結構性閘門

**✅ 升級項目 2：完成閘門必須執行**
- 無聲損傷：🔴 最高 — 跳過 = 記憶卡未更新，下一個對話繼承過期記憶，跨對話累積損傷
- HALT 觸發條件：Writer/Worker 角色工作流結案前，未見完成閘門輸出 → HALT
- 放置位置：`_completion_gate.md` 加入不可略過的 HALT 語法

**❌ 不升級：跨語系面板**
- 面板沒出現不會造成程式碼損壞（只少一次語意確認）
- 更好的方向是 D10（瘦身），降低面板認知負荷讓它更不容易被跳過

**❌ 不升級：記憶卡兩步驟流程**
- 已被完成閘門 HALT 涵蓋，不需重複設防
